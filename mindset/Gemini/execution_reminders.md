# 执行阶段核心叮嘱与路线约束 (Execution Reminders)

这份文档是对整个研究路线的“防偏题”与“防美化”约束，指导后续的工程执行与论文撰写。

## 1. 核心定位锁死：做“蒸馏保真性”，不做“人格排行榜”

我们**不是**第一个把 CoT 当图分析的团队，也**不是**在做一个玄学的“大模型 MBTI 人格排行榜”。

我们的核心独特价值在于：**公开推理蒸馏数据的结构保真性审计**。
*   **痛点**：现有的蒸馏数据往往只保留了 Teacher 的批判词汇（表层批判密度），却丢失了批判—修正—验证的拓扑连接（深层修正耦合）。
*   **方法**：用低成本、可复现的文本结构指标，审计蒸馏数据中的 correction coupling / dependency depth / inertia。
*   **价值**：人类作为认知审计者，审计训练数据中推理结构是否真的有因果闭环。

**关于模型人格（Claude/GPT/Gemini 比较）的处理：**
*   **主实验（正文）**：只做蒸馏数据中的推理拓扑保真性。
*   **文献综述/讨论**：把“模型人格”作为讲故事和提出问题的引子，将人格语言降级为认知风格解释框架，而不是直接给模型做人格诊断。

## 2. 指标落地策略：工程做全量，论文分层报告

对于 `operationalization_protocol.md` 中设计的指标，工程上可以全部实现，但在论文报告时必须克制，防止变成“指标海”。

*   **必保主指标（主实验结论）**：CCS (修正连接强度) / PDD (段落依赖深度) / IS (惯性斜率)
*   **辅助指标（解释结构差异）**：CI / RT / RAE
*   **可消融指标（后续删减、验证、附录）**：EAR / OED / BRD / bigram entropy / topology type

## 3. 防控美化风险：Anti-beautification Audit

对 `walkthrough.md` 或早期设计中的漂亮论断（如“最具原创性”、“完美适配”、“零成本”），必须设立专门的反向审计机制：
1. 数据真的支持吗？
2. 是否有反例？
3. 如果结果不显著，论文还能成立吗？

在论文中采用更稳健的表述：*“本文尝试提供一种低成本、可复现的结构审计协议，并在若干公开 CoT 蒸馏数据集上进行探索性验证。”*

## 4. 第一阶段 Minimum Viable Loop (最小验证闭环)

不要贪大求全，第一步先做一个天然具有控制变量的对照组跑通逻辑：

*   **数据选择**：DeepSeek-V4-Distill vs GLM-5.1 同题配对。因为 DeepSeek 的问题来自 GLM，这是天然的“同一问题、不同 Teacher 回答”对照，能最大限度减少领域、难度、prompt 差异。
*   **计算指标**：批判词密度 baseline + 主指标 (CCS, PDD, IS)。
*   **人工验证**：人工抽检 100 条，打上 CCR 0–4 的分数。
*   **核心检验**：看 CCS/PDD/IS 这三个计算指标，是否能有效区分出“真修正”（高 CCR）和“仪式性修正/假批判”（低 CCR）。

只要这一步跑通，论文的核心骨架（批判密度 ≠ 批判耦合）就已经立住了。
